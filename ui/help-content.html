ZX-M8XXX v0.9.6 Help

Overview

ZX-M8XXX (ZX Matrix) is a vanilla JavaScript ZX Spectrum emulator with integrated debugger for reverse engineering.

Features
  • Full Z80 CPU emulation (all documented + undocumented opcodes)
  • 48K, 128K, +2, +2A, +3, Pentagon, and Scorpion machine support
  • Memory banking (128K/+2/+2A/Pentagon/Scorpion)
  • ULA video generation with border
  • ULAplus extended palette (64 colors, HAM256 raster effects)
  • Memory contention emulation
  • AY-3-8910 sound chip (stereo modes)
  • PSG file export (record AY music)
  • SNA/Z80/SZX snapshot loading/saving
  • Quicksave/Quickload (F2/F5) - instant state save to browser storage
  • TAP/TZX tape loading (instant or real-time with border stripes and sound)
  • TRD/SCL disk images (TR-DOS)
  • DSK disk images (+3 ВµPD765 FDC)
  • ZIP archive support
  • RZX playback (partial)
  • Game Browser - search and download games from Spectrum Computing
  • Fullscreen mode (F11) with crisp/fit/stretch options
  • Automated test suite with screenshot comparison
  • Compare tool for snapshots and binaries
  • Explorer for file analysis (BASIC decoder, disassembly, hex dump)
  • Game Mapper for capturing rooms and building navigable map grids
  • OCR Text Ripper for screen text recognition and font extraction
Quick Start
  1. Place ROM files in roms/ directory (48.rom, 128.rom, plus2.rom, plus2a.rom, plus3.rom, pentagon.rom, scorpion.rom, trdos.rom)
  2. Open index.html in a modern browser
  3. Use Load в†’ Web to browse and download games online
  4. Or Load в†’ File to load SNA/Z80/SZX snapshots, TAP/TZX tapes, TRD/SCL/DSK disks, or project files (.zxproj/.json)
  5. F2 = Quicksave, F5 = Quickload during gameplay

Debugger

The built-in debugger provides comprehensive tools for development and reverse engineering.

Registers View

Shows all Z80 registers including the alternate set (AF', BC', DE', HL'). Flags are displayed as visual indicators: S, Z, H, P/V, N, C.

О”T (Delta T-states): Accumulated T-states since the last breakpoint fired. Useful for measuring cycle-exact timing between breakpoints. The counter accumulates through steps, runs, and all execution modes. Resets to 0 when any breakpoint fires (exec, memory, or port).

Stack View

Shows current stack contents from SP upward. Changed values are highlighted. Click an entry to navigate to that address (prefers disasm panel, falls back to memory view). Right-click for context menu with explicit panel choices.

Calls Panel

Runtime call stack tracking. Monitors CALL, RST, and INT instructions during execution and maintains a live subroutine chain. When RET/RETI/RETN executes, the corresponding entry is removed. Most recent call is shown at the top.

  • Click: Navigate to address (prefers disasm panel, falls back to memory)
  • Right-click: Context menu for explicit panel choice (disasm/memory, left/right)
  • INT marker shown for interrupt-triggered entries
  • Labels from debugger and assembler are displayed alongside addresses
  • Direct SP manipulation (LD SP,nn etc.) resets the call stack
  • POP instructions do not affect the call stack (only true RET/RETI/RETN)
  • Cleared on machine reset; max depth: 32 levels

Signature Packs

Modular knowledge base for automatic label and region recognition. Each pack is a self-contained JSON file with labels, regions, and byte-pattern anchors for matching.

Pack Types
  • Engine packs — Match known engines (AGD, Quill, PAW, etc.). One pack covers all games using that engine version.
  • Game packs — Labels for a specific game from community disassemblies.
Workflow
  1. Import a .skool file, .asm/.a80 source, .zip archive, or pack JSON via Settings → Signatures
  2. Build Anchors — with the game loaded, click "Anchors" to snapshot byte patterns at key labeled addresses. Anchors automatically mask 16-bit address operands (CALL nn, JP nn, LD rr,nn, etc.) for cross-game matching.
  3. Scan Memory — load a game and click "Scan Memory" to find matching packs. Results show confidence percentage. Two-pass matching: exact first, then masked fallback.
  4. Apply — click "Apply" on a match to add labels and regions. Per-anchor offset voting handles procedures relocated to different addresses.
  5. Direct Apply — click "Apply" in the pack list to apply all labels at offset 0 (no scanning needed).
Cross-Game Matching

Signature packs can match code across different games sharing the same engine (e.g. applying EATF labels to Bomberman). Each procedure is matched independently with its own offset, so shuffled code is handled correctly. Address operands in Z80 instructions are masked automatically, since absolute addresses differ between builds.

Supported Formats
  • .skool (SkoolKit) — reads @label= directives, control chars (c=code, b=data, t=text, w=words), inline comments
  • .asm / .a80 (sjasmplus, pasmo, z80asm) — reads labels, ORG directives, EQU constants, data directives (DEFB/DEFW/DEFS/DEFM). INCLUDE directives resolved when importing multiple files.
  • .zip — archives containing any combination of .skool, .asm, and .a80 files. All source files are extracted and processed together.
GitHub Browser

Click GitHub to browse a repository. Paste a URL (e.g. github.com/Bedazzle/EATF) or owner/repo shorthand. The scanner finds .skool, .asm, .a80 files recursively (up to 3 levels). ASM files are downloaded as a batch so INCLUDE directives resolve across the project.

Memory Breakpoints

Break on memory access:

  • Break on Read: Pause when address is read
  • Break on Write: Pause when address is written
Port Breakpoints

Break on I/O port access (IN/OUT instructions).

Screen Region Breakpoints

Break on writes to rectangular screen regions:

  • Write Bitmap: Break when pixel data is written to the specified region
  • Write Attr: Break when attribute data is written to the specified region

Enter coordinates as C,R,W,H (column, row, width, height in character cells 0-31, 0-23). For bitmap triggers, tick the px checkbox for pixel coordinates (X,Y,W,H with x 0-255, y 0-191).

The Screen dropdown selects the target: Normal (page 5 at $4000), Shadow (page 7 at $C000), or Both.

Tape & Disk Triggers
  • Tape Block: Break after a tape block is loaded (flash or real-time). Use the skip count to break after N blocks (e.g. skip=4 breaks after the 5th block).
  • Disk Read: Break after any disk sector read (TR-DOS or +3DOS). Use skip count to break after N reads.
  • Disk Sector: Break when a specific track:sector is read. Enter as TT:SS (e.g. 0:1 for track 0, sector 1).
Port I/O Log

In the Trace tab, enable Port I/O logging to record all port reads/writes. Export produces a TSV file with columns: Dir, Port, Value, PC, Src, Frame, T-states.

The Src column identifies the ROM source when PC is in 0000-3FFF:

  • TRDOS — TR-DOS ROM is paged in (Beta Disk)
  • ROM:N — ROM bank N (e.g. ROM:0 = 128 BASIC, ROM:2 = +3DOS)
  • RAM — RAM mapped over ROM (Pentagon 1024, Scorpion, +2A special paging)
  • empty — PC is in RAM (address ≥ 4000h)

POKE Search

Find memory locations storing game values (lives, score, etc.) by comparing snapshots:

  1. Snap — take a memory snapshot (status shows snap count)
  2. Play game until the target value changes (e.g. lose a life)
  3. Snap again — repeat snap/play for more data points
  4. Search — scans all RAM, validates every snap-to-snap transition:
    • -1 / +1 — value changed by exactly 1 between each snap
    • Decreased / Increased — value went down/up between each snap
    • Changed — value differs between each snap
    • Unchanged — value stayed the same across all snaps
  5. Take more snaps and search again to narrow candidates
  6. Filter — enter a hex value to keep only candidates where current memory matches. Clear the value and click Filter to undo
  7. Click result to view in memory dump; hover to see value at every snap point

Requires at least 2 snapshots. Each search always scans all RAM — more snaps = fewer candidates. No deduplication: every snap-to-snap pair must match the selected mode.

Skip screen (on by default) — excludes screen memory (4000-5BFF) to reduce false positives.

Write Monitor

Monitors writes to a specific address to find collision detection routines. Typical workflow:

  1. Use POKE Search to find the lives/health address
  2. Enter the address in the Watch field ($hex, 0xhex, or decimal)
  3. Click Monitor — the button turns active and monitoring begins
  4. Play the game — trigger a collision or death
  5. Click Stop — results appear grouped by writing instruction

Each result shows:

  • The writing instruction (PC + disassembled mnemonic) with value transitions
  • Call chain — the full call stack at the time of the write
  • NOP suggestion — identifies the CALL instruction to patch out

All addresses are clickable and navigate to the disassembler. Monitoring stops automatically on machine change or reset.

Command-Line Defines

Use the Defines input field to pass defines for conditional assembly:

  • Format: NAME,NAME=value,...
  • Simple define: DEBUG (value = 1)
  • With value: VERSION=5 or BUILD=$100
  • Multiple: DEBUG,VERSION=5,RELEASE
@define Markers

Add ; @define comments in the first 50 lines of your main file to show a dropdown of available defines:

; @define DEBUG
; @define RELEASE
; @define VERSION=5
; @define BUILD=$0100

The dropdown appears automatically. Ctrl+click to select multiple defines.

Output Files

Use directives to generate output files:

  • SAVEBIN "file.bin", start, length - Save binary data
  • SAVESNA "file.sna", start - Save 48K SNA snapshot
  • SAVETAP "file.tap", CODE, "name", start, length - Save TAP file
  • EMPTYTAP "file.tap" - Create empty TAP (for multiple blocks)

Generated files are listed in the output with size and MD5 hash.

MD5 Checksum Verification

Add expected MD5 hash in a comment to verify output:

SAVEBIN "game.bin", $8000, $1000  ; md5: a1b2c3d4e5f6...

Or use the MD5CHECK macro:

MD5CHECK "game.bin", "a1b2c3d4e5f6..."

Output shows MD5 OK or MD5 MISMATCH.

Dual-Panel Editor (Edit tab)

The Edit tab uses a two-panel layout inspired by orthodox file managers (Norton Commander, Total Commander). Each panel independently holds a file in any supported format — TAP, TZX, TRD, SCL, or DSK. The toolbar between the panels adapts to the active panel's format.

  • Active panel - Click a panel to make it active. The toolbar, Load File button, and New dropdown all operate on the active panel
  • New - Dropdown to create an empty file in the active panel: New TAP, New TZX, New TRD, or New DSK
  • Load File - Loads a file into the active panel (and into the Explorer for analysis). Supports loading from local files and from ZIP archives
  • Copy » / « Copy - Copy selected blocks/files from the active panel to the other panel. Works across different formats — tape blocks can be copied to disk and vice versa, with automatic format conversion (header fields mapped to disk attributes, filenames truncated to fit). If the destination panel is empty, a matching format is auto-created
  • Link Header - For tape panels (TAP/TZX): auto-selects header+data block pairs together for all operations
TAP Editor (Edit tab)

Edit TAP tape files: add, delete, reorder blocks, rename files, and save modified tapes.

Edit TZX tape files with full round-trip support. Standard speed blocks (ID 0x10) use the same editing workflow as TAP. Non-standard blocks (turbo, pure tone, metadata, etc.) are shown as dimmed info rows and preserved on save.

Game Mapper

The Mapper tab (Utils → Mapper) captures game screens and stitches them into a navigable map grid. Useful for documenting game layouts, creating maps, and reverse engineering room structures.

Basic Workflow
  1. Navigate to a room in the game
  2. Click Capture (or Ctrl+Space) to capture the screen
  3. Use arrow buttons (or Ctrl+Arrow keys) to move to an adjacent grid position
  4. Navigate to the next room in the game and capture again
  5. Repeat until the map is complete
Toolbar
  • Capture — capture the current screen (or configured sub-region) into the current room slot
  • Arrow buttons — move the current room position on the grid
  • Room label — shows current (X, Y) grid coordinates
  • Floor buttons (F: ▼ / ▲) — navigate between floors for multi-level games
  • Blend — per-pixel mode blend of all screenshots (picks most frequent color at each pixel, extracting static background and discarding moving sprites). Appears when room has 2+ screenshots
  • Stamp — copy a rectangular region from the selected screenshot onto the blended image to correct blend artifacts (appears when blend exists and a screenshot is selected). Opens a dialog over the overview — drag to select a rectangle, pixels are stamped immediately. Multiple stamps supported; Escape to close
  • Mark color — mark the room with a colored diagonal cross (X) overlay using one of the 8 ZX Spectrum colors. Marks appear in the overview, hover popup, and PNG export. Select “No mark” to remove
  • Delete — delete the currently selected screenshot
  • Thumbnails — clickable strip of all screenshots for the current room; click to select which is displayed
  • ⚙ (gear) — toggle the settings panel
Settings Panel
  • Region (X, Y, W, H) — capture sub-region in character cells (default: full screen 0,0,32,24). Locked once any screenshot is captured to prevent dimension mismatches; clear the map to change
  • Highlight — color used to outline the current room in the overview
  • Gap H / V — horizontal and vertical pixel spacing between rooms in the overview and export (useful for games with non-connected rooms or varying row/column spacing)
  • Zoom — overview zoom level: Fit (auto-scale to container), x1 (native pixel size), x2 (double size). x1 and x2 enable scrolling when the map exceeds the container
  • Follow — when enabled in x1/x2 zoom mode, the overview auto-scrolls to keep the current room visible
  • Export — multi-floor PNG export layout: Separate PNGs (one file per floor), or composite grid (N×... for column-count, ...×N for row-count). Empty floors are skipped
  • Floor gap — pixel spacing between floors in composite PNG exports
  • Game / Level / Author — metadata fields saved in the JSON file
  • Save — export the map as a JSON file
  • Load — import a previously saved JSON map
  • Export PNG — export the full stitched map as a PNG image
  • Clear — remove all rooms from the map
Map Overview

The overview canvas shows all rooms on the current floor stitched together. In Fit mode (default), auto-scaled to the available space. In x1/x2 mode, rendered at native/double pixel size with scrollbars; enable Follow to auto-scroll to the current room. The current room is highlighted with a colored border.

  • Click a room in the overview to select it
  • Hover over a room to see a 2x zoom popup preview
Multi-Floor Support

Games with multiple levels or floors can be mapped using the floor dimension. Each floor is an independent 2D grid. The overview shows only the current floor. Use the floor buttons or Ctrl+Shift+Up/Down to switch floors.

OCR Text Ripper

Screen text recognition tool. Captures ZX Spectrum screen pages, identifies character glyphs by pixel hash, and maps them to text characters. Supports custom cell sizes (4–8 × 4–16 pixels) for non-standard fixed-width fonts. Access via the OCR tab (Utils → OCR).

Basic Workflow
  1. Set CW/CH to match the font’s character dimensions (default 8×8). Use the Grid overlay to visually align the grid to the font on screen
  2. Click Capture to grab the current screen as a page of glyph tiles
  3. Unrecognized glyphs appear in the grid — click a glyph tile to assign a character
  4. Previously mapped glyphs are recognized automatically on subsequent captures
  5. The text output panel shows the recognized text

The region spinners (X/Y/W/H) and maximum values adjust automatically to the cell grid dimensions.

Preview Zoom

Three zoom buttons (x1/x2/x3) below the preview canvas. Default is x2. The grid overlay renders at display resolution for crisp lines at any zoom level.

Capture Known Mode

The Known checkbox next to Capture: when enabled, only glyphs already mapped in any charset are captured. Unknown glyphs are replaced by spaces in the text output. Useful for ripping text from screens with mixed text and graphics.

Charsets

Glyph-to-character mappings are stored in charsets. Each charset is an independent mapping table with its own CW/CH/OX/OY/GX/GY settings.

  • Create multiple charsets for different fonts used in the same program
  • Text recognition searches all charsets (active charset first, then others)
  • Glyph editing (click-to-assign) applies to the active charset only
  • Use Save / Load to export/import charsets as JSON files
Bulk Glyph Delete

Ctrl+click glyph tiles to toggle multi-selection (orange highlight). A Del N selected button appears in the glyph header — click it to remove all selected glyphs from the charset at once.

  • Use the Graphics Viewer (Char mode) to locate custom fonts in memory, then click →OCR to extract directly
  • The GFX viewer’s Bank selector lets you browse extended RAM banks (128K) — the selected bank is passed to the Extract Font dialog
  • Multiple extractions accumulate in the same charset (new hashes are added, existing ones updated)
  • After extracting, capture a screen — ROM font characters are recognized immediately
  • For programs with two different fonts, extract each into a separate charset — recognition uses all charsets automatically

  • Enable Grid overlay and adjust CW/CH/OX/OY/GX/GY to match the font’s cell boundaries visually before capturing
  • Use Known mode to capture only text without polluting the charset with graphics glyphs

POKE Manager

The Pokes tab in the debugger panel manages named pokes (multi-address byte patches) and memory value editors. Two-column layout: editors on the left, pokes on the right. Create entries inline or load from JSON files.

Layout

Editors column (left) and pokes column (right) are shown side by side. Load, Save, and Clear buttons are at the top right. Click the game name label to edit it.

Adding Pokes

Use the add form at the top of the pokes column: enter a name, address, original value, and poke value, then click +. If a poke with the same name already exists, the new patch is added to it (multi-patch building). All addresses and values are hex.

Poke Entries

Each poke has a checkbox to toggle it on/off. Enabling writes the patched bytes to memory; disabling restores the original values. Changes are instant — no apply button needed. The master checkbox above the poke list enables or disables all pokes at once (indeterminate state when partially enabled). Click × to remove an entry (restores original bytes if enabled).

Adding Editors

Use the editor add form at the top of the editors column: enter a name, address, and value type (byte or word), then click +. Read refreshes all editor values from memory.

Memory Editors

Editors provide live read/write access to specific memory addresses. All values are hex. Editing a value and pressing Enter or leaving the field writes it to memory immediately. Click × to remove an editor.

Word editors use little-endian byte order (low byte at address, high byte at address+1).

JSON File Format

Create poke files in a text editor. All addresses and values are hex. Patches use compact [addr, normal, poke] arrays.

{
    "game": "Game Name",
    "pokes": [
        {
            "name": "Infinite lives",
            "enabled": true,
            "patches": [["8C2A", "3D", "00"]]
        },
        {
            "name": "Walk through walls",
            "patches": [["8E01", "28", "18"], ["8E10", "3C", "00"]]
        }
    ],
    "editors": [
        { "name": "Lives", "addr": "8C2B", "type": "byte" },
        { "name": "Score", "addr": "8C30", "type": "word" }
    ]
}
Fields
  • game — optional display label shown in the toolbar
  • pokes[].name — display name for the poke entry
  • pokes[].enabled — initial enabled state (boolean, default false)
  • pokes[].patches[] — arrays of [addr, normal, poke]
  • editors[].name — display label for the editor
  • editors[].addr — memory address
  • editors[].type — "byte" or "word" (little-endian)
Project Persistence

Pokes and editors are saved and restored with the project file, including which pokes are currently enabled.

Known Limitations
  • Chained prefix interrupts: On real Z80, interrupts can occur between chained DD/FD prefix bytes (each prefix is a separate instruction fetch). This emulator processes prefix chains atomically within a single step, so interrupts cannot occur mid-chain. This rarely affects real software.
  • Floating bus: Not fully emulated. Some copy protection schemes may not work.
  • Z80 save uncompressed: Z80 snapshots saved without RLE compression for compatibility (~131KB for 128K, ~49KB for 48K).
